# v0.1 原生桌面端 AI 创作平台框架版 QA 验收清单

适用版本：Endless Creation v0.1 framework slice  
定位：原生桌面端（Electron + React）AI 创作平台框架版。  
验收重点：只验证桌面壳、renderer、bridge/fallback、基础交互与框架可扩展性；不验收真实业务能力。

## 0. 验收范围与原则

v0.1 的目标是把产品收敛为“可启动、可构建、可交互、可安全扩展”的桌面端框架，不继续扩业务功能。

QA 只关注：

- 构建命令稳定。
- Electron shell 可启动并加载 React renderer。
- 浏览器预览仍可作为 renderer fallback 运行。
- 主题切换、导航、mock 生成等框架级交互不退化。
- preload bridge / renderer fallback 边界清晰。
- 项目库 adapter / fallback 的框架路径可被验证，不要求真实持久化或复杂资产管理。

## 1. v0.1 框架验收项总表

| 模块 | 验收项 | 通过标准 |
| --- | --- | --- |
| 启动 | `npm run dev:electron` | Electron 窗口启动，renderer 正常加载，无白屏 |
| 构建 | `npm run build` | renderer 与 Electron TypeScript 构建均通过 |
| Electron shell | BrowserWindow | 默认尺寸合理，最小尺寸限制有效，关闭窗口进程正常退出 |
| Preload bridge | bridge API | renderer 可访问白名单 bridge，不暴露 Node 原始能力 |
| Renderer fallback | `npm run dev` / `npm run preview` | 浏览器中可运行，不依赖 Electron bridge |
| 主题切换 | dark/light | Electron 与浏览器中均可切换并持久化 |
| 导航 | active mode | 文本/图片/视频/项目库导航可切换，active 状态唯一 |
| Mock 生成 | 校验/loading/success/error/retry/copy | 仅 mock，本地可重复测试，无真实 AI 依赖 |
| 项目库 adapter/fallback | 框架边界 | 项目库入口可用；adapter 不可用时 fallback 不崩溃、有明确占位或本地 mock 行为 |
| 安全边界 | Node 暴露检查 | renderer 不能直接访问 `require`、`fs`、`child_process` |

## 2. 构建与命令测试

### TC-01：安装依赖后构建

步骤：

```powershell
npm ci
npm run build
```

预期结果：

- `build:renderer` 通过。
- `build:electron` 通过。
- 生成 `dist/` 与 `dist-electron/`。
- 无 TypeScript error。
- 无 Vite build error。

### TC-02：轻量检查命令

步骤：

```powershell
npm run check
```

预期结果：

- `check` 命令存在。
- 当前 v0.1 可等价执行 `npm run build`。
- 失败时应阻塞合并。

### TC-03：桌面开发启动

步骤：

```powershell
npm run dev:electron
```

预期结果：

- Vite renderer dev server 启动。
- Electron main 编译并启动。
- 桌面窗口出现。
- 终端无 main process crash。
- DevTools Console 无 renderer runtime error。

## 3. Electron shell 手动验收

### TC-04：窗口启动与加载

步骤：

1. 执行 `npm run dev:electron`。
2. 等待窗口显示。
3. 观察首屏。

预期结果：

- 窗口标题为 Endless Creation 或约定标题。
- React renderer 正常显示。
- 首屏包含导航、主题按钮、Prompt 输入区、结果/项目区域。
- 无白屏、无 `Failed to load URL`、无 `ERR_FILE_NOT_FOUND`。

### TC-05：窗口基础行为

步骤：

1. 最小化窗口。
2. 恢复窗口。
3. 最大化窗口。
4. 还原窗口。
5. 关闭窗口。

预期结果：

- 所有窗口操作无崩溃。
- 关闭窗口后进程按平台约定退出或进入后台。
- 再次运行 `npm run dev:electron` 可正常启动。

### TC-06：安全配置人工检查

步骤：

1. 打开 Electron DevTools Console。
2. 执行：

```js
typeof require
typeof window.require
typeof process
typeof Buffer
```

预期结果：

- `require` 与 `window.require` 不可用。
- 不可通过 renderer 直接访问 `fs`、`path`、`child_process`。
- 如果存在有限 `process` 信息，也不得提供文件系统或进程执行能力。
- 所有本地能力必须通过 preload bridge 白名单 API。

## 4. Preload bridge / renderer fallback 验收

### TC-07：Electron bridge 可用

步骤：

在 Electron DevTools Console 执行：

```js
typeof window.endlessCreationBridge
Object.keys(window.endlessCreationBridge || {})
```

预期结果：

- bridge 对象存在。
- 仅暴露白名单能力，例如 `app`、`window`、`clipboard`。
- 不暴露 `ipcRenderer` 原始对象。
- bridge 方法调用失败时不导致 renderer 白屏。

### TC-08：浏览器 fallback 可运行

步骤：

```powershell
npm run dev -- --host 127.0.0.1 --port 5173
```

打开：

```text
http://127.0.0.1:5173/
```

预期结果：

- 浏览器中 renderer 正常显示。
- `window.endlessCreationBridge` 可以不存在。
- 主题切换、导航、mock 生成不因缺少 Electron bridge 崩溃。
- 复制能力若受浏览器权限限制，应显示合理失败反馈。

### TC-09：生产预览 fallback

步骤：

```powershell
npm run build
npm run preview
```

预期结果：

- 浏览器中可打开生产构建预览。
- 不依赖 Electron main/preload。
- 无 runtime error。

## 5. 主题切换验收

### TC-10：Electron 中主题切换

步骤：

1. 启动 `npm run dev:electron`。
2. 点击主题切换按钮。
3. 切换到浅色。
4. 再切回深色。
5. 刷新 renderer 或重启应用。

预期结果：

- 主题即时变化。
- 文案、边框、按钮状态可读。
- 主题持久化按当前实现生效。
- 重启或刷新后不出现错误主题闪烁导致不可读。

### TC-11：浏览器中主题切换

步骤：

1. 启动 `npm run dev`。
2. 在浏览器打开页面。
3. 重复 TC-10。

预期结果：

- 与 Electron 中表现一致。
- Web fallback 下 localStorage 可用。
- 不依赖 preload bridge。

### TC-12：active 导航切换

1. 点击 `文本生成`。
2. 点击 `图片生成`。
3. 点击 `视频生成`。
4. 点击 `项目库`。
5. 使用键盘 Tab + Enter/Space 重复上述操作。

预期结果：

- 键盘可聚焦并触发导航。
- 不出现 Console error。

### TC-13：导航影响当前工作台语义

步骤：

1. 切换到每个导航项。
2. 观察标题、说明、Prompt placeholder 或生成模式文案。

预期结果：

- 当前工作台语义与导航一致。
- 项目库入口可进入或显示对应框架占位。
- 不要求真实项目库业务功能。

## 7. Mock 生成验收

### TC-14：Prompt 校验

步骤：

1. 清空 Prompt。
2. 输入纯空格。
3. 点击生成。

预期结果：

- 显示校验提示。
- 不进入 loading。
- 不创建 mock 任务。
- 输入有效文本后校验提示消失。

### TC-15：loading / success / error / retry

步骤：

1. 输入有效 Prompt。
2. 点击生成。
3. loading 期间快速重复点击。
4. 等待 mock 结果。
5. 多次生成直到出现 success 与 error。
6. error 时点击重试。

预期结果：

- loading 期间按钮禁用或阻止重复提交。
- success 展示 mock 标题、摘要和正文。
- error 展示错误提示与重试入口。
- retry 使用当前 Prompt 与当前导航模式。
- 全流程不依赖真实 AI 服务。

### TC-16：复制反馈

步骤：

1. 生成成功结果。
2. 点击复制结果。
3. 粘贴到文本编辑器验证。

预期结果：

- Electron 中优先走 bridge clipboard。
- 浏览器中走 Web clipboard fallback。
- 成功时显示已复制反馈。
- 失败时显示可理解失败反馈，不崩溃。

## 8. 项目库 adapter / fallback 验收

v0.1 只验收项目库作为框架入口与 adapter/fallback 边界，不验收复杂资产管理。

### TC-17：项目库入口

步骤：

1. 点击 `项目库` 导航。
2. 观察工作台标题、说明、Prompt placeholder 或占位区。

预期结果：

- UI 明确进入项目库语义。
- 不出现空白页面。
- 不出现与文本/图片/视频模式混淆的 active 状态。

### TC-18：adapter 不可用时 fallback

步骤：

1. 在浏览器 fallback 中打开项目库。
2. 在 Electron 中打开项目库。
3. 如存在 adapter 状态提示，观察其状态。

预期结果：

- adapter 不存在或不可用时，UI 使用本地 mock / 静态数据 / 占位说明。
- 不抛出未捕获异常。
- 不要求真实文件数据库、真实云端项目、真实素材索引。
- 用户能理解当前为框架版或 mock/fallback 状态。

### TC-19：项目库 mock 生成

步骤：

1. 切换到 `项目库`。
2. 输入整理类 Prompt。
3. 点击生成。

预期结果：

- mock 结果体现项目库整理语义。
- 不访问真实文件系统或云服务。
- 失败/重试/复制反馈与其他模式一致。

## 9. 响应式检查点

每个宽度均需在 Electron 窗口和浏览器预览中检查。若 Electron 设置了最小窗口宽度，低于最小宽度的检查可在浏览器预览中完成，并记录 Electron 最小尺寸限制。

### 320px

检查点：

- 浏览器预览可缩至 320px。
- 页面整体不应出现不可控水平滚动。
- 导航横向滚动可接受，但必须可操作。
- Prompt、生成按钮、错误提示、复制反馈不被截断。
- 结果正文自动换行。

### 768px

检查点：

- 导航与内容布局合理折行。
- Prompt 区与结果区不重叠。
- 快捷卡片/面板布局不挤压文字。
- 主题按钮和项目库入口可见可操作。

### 1024px

检查点：

- 桌面主布局可用。
- 侧边导航或主区域比例合理。
- 生成结果区域可读。
- loading/error/success 状态不造成布局跳动过大。

### 1440px

检查点：

- 最大宽度约束或桌面留白合理。
- 内容不过度拉伸。
- 视觉层级清晰。
- Electron 最大化后无大面积异常空白或错位。

## 10. 明确不应测试 / 不应验收的非目标

v0.1 framework slice 不验收以下内容：

- 真实 AI 生成质量。
- 真实 AI API 调用、API key 管理、模型选择、计费。
- 真实账号系统、登录、注册、权限、组织空间。
- 云同步、多端同步、远程项目存储。
- 复杂资产管理：素材库索引、批量导入、标签体系、版本管理。
- 真实文件数据库或生产级本地持久化。
- 视频渲染、图片生成、长文编辑器等业务深功能。
- 插件市场、工作流编排、多用户协作。
- 自动更新、安装包签名、发布渠道。

如果测试中发现上述功能缺失，不应作为 v0.1 缺陷；只可记录为后续版本需求。

## 11. 后续自动化测试建议（仅建议，不在本任务引入）

建议后续按阶段补充，不在 v0.1 直接引入大型测试框架：

1. 轻量命令门禁：保持 `npm run check` 覆盖 build。
2. Renderer 单元测试：后续可考虑 Vitest + React Testing Library，覆盖主题、导航、Prompt 校验。
3. Electron smoke test：后续可考虑 Playwright Electron 或 Spectron 替代方案，验证窗口启动、bridge、Node 不暴露。
4. 可访问性测试：后续可加入 axe 检查，覆盖按钮名称、aria-current、aria-invalid、焦点路径。
5. 视觉回归：后续对 320/768/1024/1440 的 dark/light 截图做基线。
6. Bridge contract test：后续为 preload bridge 增加契约测试，确保只暴露白名单 API。
